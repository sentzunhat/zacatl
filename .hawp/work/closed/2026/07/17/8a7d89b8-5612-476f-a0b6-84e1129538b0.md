# Default security headers, opt-in CORS/rate-limit, configurable host binding

## Work Item

**UUID:** `8a7d89b8-5612-476f-a0b6-84e1129538b0`
**Type:** security
**Priority:** P2
**Reported:** 2026-07-13
**Source:** Security/arch/scalability audit — `.hawp/work/evidence/2026/07/13/AUDIT-002-security-arch-scalability-report.md`

## Investigation

Confirmed LOWs (adversarially verified):

- No security-header, CORS, or rate-limit layer is registered by the framework
  (`src/service/platforms/server/api/api-server.ts:39-53`) — services are
  insecure-by-default; every consumer must hand-roll HSTS/CSP/rate limiting.
- Host is hardcoded to `0.0.0.0` in both providers
  (`fastify/api-adapter.ts:54`, `express/api-adapter.ts:125`) — loopback-only
  binding is impossible, and Fastify's safer localhost default is overridden.

## Plan

1. Register a helmet-equivalent security-header set by default
   (@fastify/helmet for Fastify, helmet for Express) with a config switch to
   disable/customize.
2. Expose opt-in CORS and rate-limit configuration in `HttpServerConfig`,
   wiring the provider-native plugins.
3. Add optional `host` to `HttpServerConfig`/`listen()`, defaulting to the
   current `0.0.0.0` for back-compat, documented.
4. Update examples and docs.

## Risk + Review Gate

**Risk:** medium — default headers can change response behavior for existing
consumers (CSP especially); consider default-on with permissive CSP or
default-off + loud docs. Decide at review.
**Gate:** plan-ready; explicit approval required before implementation.

## Verification

Header assertions in provider tests for both Express and Fastify, host-binding
test (loopback), example smoke tests, full suite green.
