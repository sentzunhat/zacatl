# Fix eager repository resolution breaking node:sqlite startup

## Work Item

**UUID:** `2898d74c-2801-44b2-936e-44aa89fda4a8`
**Type:** bugfix
**Priority:** P0
**Reported:** 2026-07-13
**Source:** Security/arch/scalability audit — `.hawp/work/evidence/2026/07/13/AUDIT-002-security-arch-scalability-report.md`

## Investigation

Confirmed HIGH (adversarially verified). `Service` constructs `Layers` eagerly
(`src/service/service.ts:43-51`), and `Infrastructure` resolves repositories at
construction (`src/service/layers/infrastructure/infrastructure.ts:40`). The
node:sqlite adapter service-locates its database instance in its constructor,
but that instance is only registered later by `connect()`. Result: the
documented node:sqlite flow throws at Service construction, before `connect()`
can ever run. The primary documented SQLite path cannot start.

## Plan

1. Reproduce with a minimal Service + node:sqlite repository (add a failing test).
2. Choose fix: (a) resolve the ORM model lazily on first query inside the
   adapter, or (b) reorder bootstrap so database registration precedes Layers
   construction. Prefer (a) — smaller blast radius, aligns with the lazy-loading
   design in `docs/third-party/orm/orm-lazy-loading.md`.
3. Apply the same lazy-resolution pattern to mongoose/sequelize adapters for
   consistency (they share the constructor service-location pattern).
4. Update adapter unit tests; add a regression test for construct-before-connect.

## Risk + Review Gate

**Risk:** medium — touches adapter construction order on the primary persistence path.
**Gate:** plan-ready; explicit approval required before implementation.

## Verification

New regression test (Service + node:sqlite constructed before connect), full
test suite, type check, lint, example smoke test (`fastify-sqlite-react`).
