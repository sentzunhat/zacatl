# Separate safe public errors from diagnostic serialization

## Work Item

**UUID:** `d5be8edb-0b18-4fb5-9de9-20ab3b5fe400`
**Type:** security
**Reported:** 2026-07-13
**Closed:** 2026-07-13

## Investigation

`CustomError.toJSON()` emitted stacks, nested errors, metadata, reasons, component
names, operations, timestamps, internal IDs, and internal messages. A shipped
Fastify example also sent its raw error object from the global handler.

## Decision

Default JSON serialization is a public boundary. It returns only a safe message,
correlation ID, and optional code. Numeric 5xx messages are normalized to
`Internal Server Error`. Full context requires explicit `toDiagnosticJSON()` use.

## What Changed

- Added exported `PublicErrorJSON` and `DiagnosticErrorJSON` contracts.
- Reduced `toJSON()` to the concise public shape and redacted 5xx messages.
- Added `toDiagnosticJSON()` for trusted internal logging.
- Changed all eight example handlers to explicitly construct safe responses and
  suppress generic 5xx details.
- Removed raw error sending from the Fastify SQLite React example.
- Updated API, architecture, testing, and migration documentation.
- Added secret, absolute-path, stack, nested-error, and diagnostic retention tests.

## Outcome

Normal JSON serialization no longer discloses diagnostic internals. Consumers that
previously used verbose `toJSON()` for logging must migrate to `toDiagnosticJSON()`.
