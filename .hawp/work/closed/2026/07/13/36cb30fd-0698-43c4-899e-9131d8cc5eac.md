# Review and align object type shapes across zacatl

## Work Item

**UUID:** `36cb30fd-0698-43c4-899e-9131d8cc5eac`
**Type:** review
**Priority:** P2
**Reported:** 2026-07-13

## Motivation

The framework has grown several distinct config/shape families
(`DatabaseConfig`, `DatabaseConnection`, `ConnectionRef`, `BaseRepositoryConfig`,
`ServiceConfig`, `ServerConfig`, `EntryPointConfig`, etc.) across multiple
layers. As `7956fd0b` introduces `DatabaseConnection` and `ConnectionRef` as
first-class objects, it is a good time to audit all public-facing object types
for consistency, naming conventions, and ergonomics before the API hardens.

## Scope

- All public config and option objects under `src/` that consumers pass at
  construction or configuration time.
- Public types re-exported from the package barrel.
- Interface vs type alias usage — prefer `interface` for extensible objects,
  `type` for unions and aliases.
- Naming: `*Config` for constructor/bootstrap options, `*Options` for
  per-call options, `*Port` for hexagonal contracts, `*Ref` for
  lightweight references to already-registered resources.
- Field naming: consistent use of objects over flat primitives where grouping
  adds clarity (e.g. `connection: { url, name? }` rather than
  `connectionString + connectionName`).
- Optional vs required fields: required fields that have safe defaults should
  be optional with documented defaults; required fields with no sensible
  default should stay required.

## Plan

1. Enumerate all public-facing object types (grep public exports + barrel).
2. For each type: note shape, naming, field optionality, and any inconsistency
   with neighboring types.
3. Produce a mismatch summary and a conservative edit plan (no behavior changes).
4. Apply only low-risk, targeted type edits — rename fields, adjust optionality,
   add/remove `interface`/`type` keyword where appropriate.
5. Update docs and examples to match.

## Dependencies
