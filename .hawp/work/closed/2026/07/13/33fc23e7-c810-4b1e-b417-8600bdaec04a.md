# Deprecate legacy HMAC SHA-1 and MD5 algorithms

## Work Item

**UUID:** `33fc23e7-c810-4b1e-b417-8600bdaec04a`
**Type:** security
**Reported:** 2026-07-13

## Investigation

The public HMAC API permits SHA-1 and MD5 without warning. HMAC constructions
are not equivalent to raw hash use, but legacy algorithms are unsuitable for new
systems and may violate security policy.

**Scope extended 2026-07-13** from the security audit
(`.hawp/work/evidence/2026/07/13/AUDIT-002-security-arch-scalability-report.md`):
the module exports only `generateHmac` (`src/utils/hmac.ts:32-43`) with no
timing-safe verification helper, so consumers naturally compare signatures with
`===`, enabling a timing side-channel. Same file, same review gate — folded in.

## Plan

1. Search public docs and consumers for explicit SHA-1/MD5 use.
2. Choose deprecation-only or removal based on semver compatibility.
3. Restrict the recommended API to SHA-256 and SHA-512.
4. If compatibility is required, isolate legacy algorithms behind an explicit API.
5. Add `verifyHmac(payload, signature, secret, algorithm)` built on
   `crypto.timingSafeEqual` (length-check first), and document it as the only
   supported comparison path.
6. Update types, tests, documentation, and changelog.

## Risk + Review Gate

**Risk:** high if removed; medium if deprecated only.
**Gate:** plan-ready; explicit approval required before implementation.

## Verification
