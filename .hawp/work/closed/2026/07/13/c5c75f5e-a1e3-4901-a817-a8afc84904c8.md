# Enforce canonical command working-directory containment

## Work Item

**UUID:** `c5c75f5e-a1e3-4901-a817-a8afc84904c8`
**Type:** security
**Reported:** 2026-07-13
**Closed:** 2026-07-13

## Investigation

`cwd.startsWith(cwdPrefix)` accepted sibling prefixes and traversal paths. Direct
reproduction accepted `/safe/prefix-evil` and `/safe/prefix/../escape` for an
allowed root of `/safe/prefix`.

## Decision

Resolve relative paths from `process.cwd()`, canonicalize the configured root and
requested working directory with `realpathSync`, and use `path.relative` with
platform-native separators to require the exact root or a true descendant.

Both paths must exist so symlink resolution and containment can be verified before
the command is spawned.

## What Changed

- Replaced string-prefix matching with canonical real-path containment.
- Added rejection for missing paths whose containment cannot be verified.
- Documented relative-path, symlink, POSIX, and Windows semantics.
- Added tests for exact roots, descendants, sibling prefixes, traversal, relative
  paths, nonexistent paths, and symlink escapes.

## Residual Risk

Node's `spawn` accepts a working-directory pathname rather than an opened directory
descriptor. A local actor with permission to replace path components between
validation and spawn could attempt a time-of-check/time-of-use race. Preventing that
requires OS-level ownership and permission boundaries outside this helper.

## Outcome

The reproduced sibling-prefix, traversal, and symlink escape cases are rejected.
Normal exact-root, descendant, and relative descendant paths remain supported.
