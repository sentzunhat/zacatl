# Specialize Docker runtime dependencies by database driver

## Work Item

**UUID:** `76d1fd81-01cf-4483-b9aa-137e4408dac6`
**Type:** improvement
**Reported:** 2026-07-13
**Closed:** 2026-07-13

## Investigation

The root production install retained all optional ORM drivers and was copied as
a 118 MB expanded layer into every image. Each example already declared exactly
one required driver (`sqlite3`, `pg`, or `mongoose`).

## Plan

Add a validated `DATABASE_DRIVER`, omit optional dependencies from the root
production tree, promote only the selected driver into the container-only root
dependencies, deduplicate, prototype SQLite/PostgreSQL, then promote to all
eight Compose files only after runtime proof.

## Outcome

- `examples/Dockerfile` validates `better-sqlite3`, `mongoose`, `pg`, or `sqlite3`.
- The build retains full dependencies for compilation, then removes root
  optional dependencies, promotes only the selected driver, prunes, and dedupes.
- All eight example Compose files pass their matching driver explicitly.
- `examples/DOCKER.md` documents manual arguments, current sizes, and Docker Scout setup.

## Verification

- All eight specialized images built successfully.
- All four SQLite images returned HTTP 200 from `/api/greetings`.
- Fastify PostgreSQL returned HTTP 200 with a temporary `postgres:16` sidecar.
- Unsupported `DATABASE_DRIVER=oracle` failed the build with exit code 1.
- All eight Compose files rendered the expected build arguments.
- Expanded sizes: PostgreSQL 351 MB, MongoDB 377 MB, SQLite 389 MB.
- Exported gzip sizes: PostgreSQL 70.2 MB, MongoDB 73.8 MB, SQLite 81.4-81.6 MB.
- Docker Scout scanned all eight specialized images and detected zero vulnerabilities.
- `git diff --check` passed before closeout.
