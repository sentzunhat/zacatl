# Make command-runner policy fail closed

## Work Item

**UUID:** `9f0a878f-4d2b-4ad6-a087-747dcea4db3c`
**Type:** security
**Reported:** 2026-07-13
**Closed:** 2026-07-13

## Investigation

`runnerPolicySchema` left `allowlist` undefined and `executeCommands` defaulted
to `{}`. The validator only checked commands when a non-empty allowlist existed,
so the API described as safe by default permitted every directly executable binary.

The public implementation had no production consumers inside this repository.
Its consumers were unit tests and documentation. The scripts command runner is a
separate build-tool implementation and was intentionally excluded from this change.

## Decision

Fail command validation when `allowlist` is missing or empty. Trusted application
code can deliberately bypass executable allowlisting with
`allowUnrestrictedCommands: true`; all other policy checks remain active.

## What Changed

- Added the default-false `allowUnrestrictedCommands` policy field.
- Rejected command execution when unrestricted mode is disabled and no non-empty
  allowlist is configured.
- Updated API examples and security guidance to use explicit allowlists.
- Added regression coverage for missing and empty allowlists and trusted unrestricted mode.

## Verification

- Node.js `v26.3.0`.
- Command-runner unit suite: 26 tests passed.
- Full unit suite: 560 tests across 73 files passed.
- Source, test, and scripts TypeScript checks passed.
- Source lint passed with one unrelated existing warning and no errors.
- ESM/CJS source and scripts builds passed.
- `npm run publish:dry:ci` reached the prepublish guard and stopped because version
  `0.0.56` is already published. Packaging after that guard was not re-run with a
  new version because changing the package version was outside this work item.

## Outcome

The public command runner now fails closed by default. Arbitrary executable access
requires a conspicuous trusted-code opt-in rather than an omitted policy field.
