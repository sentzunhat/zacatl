# Replace JSONC comment-strip regex with a string-aware parser

## Work Item

**UUID:** `db4a3db2-1ae2-47e5-b049-860927d3a5e4`
**Type:** bugfix
**Priority:** P0
**Reported:** 2026-07-13
**Source:** Security/arch/scalability audit — `.hawp/work/evidence/2026/07/13/AUDIT-002-security-arch-scalability-report.md`

## Investigation

Confirmed MED (adversarially verified). The JSONC comment stripper in
`src/configuration/json.ts:11-15` is a plain regex with no awareness of string
literals. Config values containing `//` (any URL, e.g.
`"endpoint": "https://api.example.com"`) are truncated and break startup;
`/* */` sequences inside string values are silently removed and the file still
parses — silent config corruption, worse than a crash.

## Plan

1. Add failing tests: URL value, `/* */` inside a string value, real comments
   outside strings, mixed cases.
2. Replace the regex with `jsonc-parser` (zero-dep, maintained) or a small
   string-literal-aware tokenizer if avoiding the dependency is preferred.
3. Keep the Zod validation step unchanged.
4. Confirm error messages still point at the offending file path.

## Risk + Review Gate

**Risk:** low — behavior only changes for inputs the current code corrupts.
**Gate:** plan-ready.

## Verification

New JSONC edge-case tests, full test suite, type check, lint.
