# Key ORM instances and adapters by database name, not vendor

## Work Item

**UUID:** `7956fd0b-5819-4112-a7ae-260c3d53ac98`
**Type:** bugfix
**Priority:** P0
**Reported:** 2026-07-13
**Source:** Security/arch/scalability audit — `.hawp/work/evidence/2026/07/13/AUDIT-002-security-arch-scalability-report.md`

## Investigation

Confirmed HIGH (adversarially verified). ORM instances are registered keyed by
vendor enum only (`src/service/platforms/server/database/orm-instance.ts:42-55`).
Registering a second database of the same vendor silently overwrites the first
registration; the orphaned connection leaks on `disconnect()`.

## Chosen Design

**`createDatabaseToken(vendor, key) → symbol`** — a single generic factory in
`orm-instance.ts`. Works for any current or future provider; no per-vendor
token file needed.

```typescript
export const createDatabaseToken = (vendor: string, key: string): symbol =>
  Symbol.for(`db:${vendor}:${key}`);
```

### `DatabaseConfig` shape

Replace flat `connectionString: string` with a `connection` object that always
carries `url` and an optional `name` alias. No union type — always an object.

```typescript
export interface DatabaseConnection {
  url: string;     // connection string / DSN / file path / ':memory:'
  name?: string;   // human alias used as the DI token key; defaults to url
}

export interface DatabaseConfig {
  vendor: DatabaseVendor;
  connection: DatabaseConnection;   // replaces connectionString
  instance?: DatabaseInstance;
  onDatabaseConnected?: OnDatabaseConnectedFunction;
}
```

**Token key derivation** — one object access, no type narrowing needed:

```typescript
const tokenKey = config.connection.name ?? config.connection.url;
createDatabaseToken(vendor, tokenKey);
```

Two unnamed same-vendor configs on different URLs → different tokens
automatically. `:memory:` works as a key for in-process SQLite. The `name`
field is the right place to put a human alias (`'users-db'`) when the URL
contains credentials or is otherwise not suitable as a readable symbol.

### Repository config — `ConnectionRef`

A separate `ConnectionRef` object is used in repository configs to reference
an already-registered connection by name. `url` is not repeated here — it
belongs only at registration time.

```typescript
export interface ConnectionRef {
  name: string;  // matches connection.name (or connection.url) from DatabaseConfig
}
```

Each adapter repository config gains an optional `connection?: ConnectionRef`.
When absent the adapter resolves the vendor's default connection (the one whose
`connection.name ?? connection.url` equals the vendor string), keeping all
existing single-database repositories working without changes.

```typescript
export interface NodeSqliteBaseRepositoryConfig {
  type: ORMType.NodeSqlite;
  name: string;                // table name — unchanged
  connection?: ConnectionRef;  // which database connection to use; defaults to vendor default
}
```

Usage at the repo side:
```typescript
super({ type: ORMType.NodeSqlite, name: 'users', connection: { name: 'users-db' } });
```

### Back-compat

- `connectionString` callers: migrate to `connection: 'path/or/dsn'` — mechanical rename.
- Single-database users with no `name`: token key = connection string value, still unique.
- The three class tokens become re-exports of the default-vendor symbol for
  any code that imported them directly (unlikely from user code, but safe).
