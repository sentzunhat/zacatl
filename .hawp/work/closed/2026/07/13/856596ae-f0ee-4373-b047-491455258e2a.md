# Make adapter initialization awaitable and narrow Mongoose error handling

## Work Item

**UUID:** `856596ae-f0ee-4373-b047-491455258e2a`
**Type:** improvement
**Priority:** P1
**Reported:** 2026-07-13
**Source:** Security/arch/scalability audit — `.hawp/work/evidence/2026/07/13/AUDIT-002-security-arch-scalability-report.md`

## Investigation

Confirmed MED (adversarially verified), two related error-swallowing patterns:

- `2898d74c` moved init to lazy `get model()` — fire-and-forget is now
  `void this.initialize().catch(console.error)` inside the getter
  (`orm/mongoose/adapter.ts`, `orm/sequelize/adapter.ts`).
  Index/table creation failures are still swallowed, early queries still race
  index builds, and there is no readiness signal.
- Mongoose adapter blanket catches return `null`/`false` for every error,
  making connection loss and timeouts indistinguishable from "not found" —
  masks outages and defeats retry or circuit-breaker logic above.

## Plan

1. Store the init promise on first model access; expose `ready(): Promise<void>`
   so callers can await full initialization.
2. Await adapter readiness inside `Service.start()` (after `connect()`) and
   fail fast with a clear error instead of degrading silently.
3. Narrow Mongoose catches to `CastError` (invalid-id → null is correct);
   rethrow everything else.
4. Align Sequelize/node:sqlite error paths with the same policy.

## Risk + Review Gate

**Risk:** medium — errors previously swallowed will now surface; consumers with
broken indexes/connectivity will see startup failures (intended, but visible).
**Gate:** plan-ready; explicit approval required before implementation.

## Verification

Tests: init failure fails `start()`; CastError still yields null; a simulated
connection error propagates. Full test suite, type check, lint.
