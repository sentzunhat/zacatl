# DI architecture: child containers, uniform guards, constructor injection, server factory seam

## Work Item

## Investigation

Confirmed MED + LOWs (adversarially verified), all facets of the same
DI/coupling design debt:

- Global tsyringe container makes `Service` non-reentrant
  (`src/dependency-injection/container.ts:3,61-79`): two Services per process
  share and overwrite registrations; `clearContainer()` is process-global.
- Adapters previously service-located their DB instance in constructors —
  fixed by `2898d74c` (lazy `get model()`). Residual issue: resolution still
  hits the global container; two Service instances share registrations.
- `isRegistered` guard applied in Domain/Infrastructure but not Application
  (`domain.ts:24,33`, `infrastructure.ts:37` vs `application.ts:39,47`) —
  `@singleton()` classes get clobbered inconsistently.
- HTTP vendors hardcoded in `Server` (`server.ts:1-15,61-88`) — no factory
  seam mirroring `database/adapters/factory.ts`; five passthrough getters
  (`server.ts:124-155`) leak internals past port boundaries.
- Duplicated error paths in `resolveDependencies`
  (`container.ts:115-127`), one branch untested.

## Plan

1. Introduce a per-Service child container (`container.createChildContainer()`)
   with the global container as fallback for back-compat; route layer
   registration through it.
2. Constructor-inject resolved ORM instances into adapters (depends on the
   lazy-init decision in `2898d74c`).
3. Apply the `isRegistered` guard uniformly across the three layers.
4. Extract a server-adapter factory; narrow or remove the passthrough getters.
5. Consolidate `resolveDependencies` error handling; add the missing
   constructor-throw test.
