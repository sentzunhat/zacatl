# Define dev-based branching and CI/CD release workflow

## Work Item

**UUID:** `75df2542-ac31-4efd-96cc-cb15356ba36c`
**Type:** tooling
**Priority:** P1
**Reported:** 2026-07-13
**Source:** Owner decision (conversation, 2026-07-13): `dev` is the working
trunk for fast iteration; feature/work branches merge into `dev`; releases are
cut from `dev` (mechanism to be decided) — the release automation does not
exist yet.

## Investigation

Current state (direct evidence):

- Existing workflows: `cve-scan.yml`, `peer-install-check.yml`,
  `publish-dry.yml` — no release/publish workflow, no branch-protection or
  merge-gating automation.
- Publishing to npm is currently manual (`npm publish` from a local machine,
  see `754afa86` re: local token exposure — rotating that token and moving
  publish into CI with a scoped automation token or OIDC trusted publishing
  would close both items).
- `main` exists as the historical default branch; `dev` now carries audit-fix
  work 6+ commits ahead of `origin/dev`.

## Plan

1. Decide the release flow (owner decision):
   - Option A (recommended, simplest): `dev` is the trunk; tags (`v*`) cut
     releases directly from `dev`; `main` tracks the latest release for
     README/browsers. Release = tag push triggers CI publish.
   - Option B: PR from `dev` → `main` per release; publish workflow triggers
     on merge to `main`.
2. Add a `release.yml` GitHub Action: on tag push — install, lint, type:check,
   test, build, `npm publish --provenance` with an npm automation token (or
   OIDC trusted publishing) stored as a repo secret.
3. Gate `dev` with the existing checks (peer-install-check, cve-scan, tests)
   as required status checks for work-branch PRs.
4. Document the flow in CONTRIBUTING or README (work branch → PR → dev →
   tag → release).

## Dependencies

- Pairs with `754afa86` (rotate npm token / engine-strict): doing publish in
  CI makes the local-token rotation the natural cutover moment.

## Risk + Review Gate

**Risk:** low-medium — publishing automation touches npm credentials; needs
owner to create the scoped token/secret.
**Gate:** plan-ready; owner must pick Option A or B before implementation.

## Verification

Dry-run the release workflow with `npm publish --dry-run` on a test tag;
confirm required checks block a failing work-branch PR.
