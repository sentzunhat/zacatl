# v0.1.0 milestone — all platforms working to a minimal standard

**UUID:** `e7a20d15` · **Type:** milestone · **Priority:** P1 · **Reported:** 2026-07-17
**Source:** Owner direction (2026-07-17): "version 0.1.0 is where i want to
have all the providers at least working to a minimal standard — the cli, the
desktop and server — all working platforms." Version updates are heavy for
the 2–4 services consuming 0.0.x, so 0.0.57 absorbed everything non-breaking;
0.1.0 collects the breaking/expansion work.

## Definition of done (minimal standard per platform)

1. **Server** — already the mature platform: Fastify/Express adapters, three
   ORMs, page serving with CDN caching, full CRUD examples + docker smoke.
   Remaining: keep green.
2. **CLI** — a runnable minimal standard: `ServiceType.CLI` boots, registers
   commands from `layers.application.entryPoints.cli`, executes a command
   end-to-end. Needs: a real example under `examples/` (like the greetings
   apps), unit + smoke coverage, docs.
3. **Desktop** — `ServiceType.DESKTOP` boots with the Neutralino platform,
   IPC handlers wired from `entryPoints.ipc`, window config honored. Needs:
   promote the `neutralino-react-transformers-webgpu` experiment into a
   minimal supported example, coverage, docs.

## Breaking changes batched for 0.1.0

- Evaluate express/fastify (+ their proxy libs) as optional peerDependencies
  so consumers install only their chosen HTTP stack (from `f2a96e04`;
  ~199 prod packages today).
- (Done early, in 0.0.57 per owner decision — no rollout window: the
  `Config*` alias removal and `generateHmac` SHA-1/MD5 removal both
  shipped as 0.0.57 breaking changes.)

## Consumer migration story

The 2–4 in-flight services handle the `Config*` rename and sha256-only
HMAC when upgrading to 0.0.57 (changelog migration table); 0.1.0 then only
adds the HTTP-framework peer move, if it lands.
