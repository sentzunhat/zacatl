# Migration Guide: 0.0.56 → 0.0.57

Zacatl 0.0.57 keeps the package version on the 0.0.x line but changes the
consumer dependency posture in an important way: database ecosystems are now
optional peers instead of unconditional runtime installs.

The main practical outcome is that services which only use Zacatl's server,
configuration, DI, validation, localization, proxy, logging, or error utilities
no longer inherit Sequelize's nested `uuid@8.3.2` advisory chain through Zacatl.

## Who Needs This Guide

Use this guide if your service currently depends on `@sentzunhat/zacatl@0.0.56`
or earlier and one of these is true:

- `npm audit --omit=dev` reports an advisory through
  `@sentzunhat/zacatl -> sequelize -> uuid`.
- Your service imports Zacatl but does not intentionally use Sequelize.
- Your service does use Zacatl's Sequelize adapter and needs to keep SQL
  persistence working after the upgrade.
- Your service is moving to Zacatl's Node 26 + distroless Docker posture.

## Runtime Contract

Zacatl 0.0.57 requires:

```json
{
  "engines": {
    "node": ">=26.0.0",
    "npm": ">=11.0.0"
  }
}
```

Node 24 is not part of the supported runtime contract for this release. Use a
Node 26 runtime locally, in CI, and in production images before treating build or
runtime failures as Zacatl defects.

## What Changed

### Sequelize Is Optional

Before 0.0.57, consumers could receive Sequelize as part of Zacatl's installed
runtime tree. That meant a non-SQL service could still inherit:

```text
@sentzunhat/zacatl
  -> sequelize
    -> uuid@8.3.2
```

In 0.0.57, Zacatl declares database packages as optional peers:

- `mongoose`
- `mongodb`
- `sequelize`
- `sqlite3`
- `pg`

Install only the packages your service actually uses.

This also changes how to read audit output:

- Non-SQL consumers should not install `sequelize`. If `sequelize` appears in a
  non-SQL application's production tree, remove the app-level dependency or find
  the package that still pulls it in.
- SQL consumers intentionally install `sequelize`, so Sequelize's transitive
  audit surface belongs to that SQL dependency choice rather than Zacatl core.
- Local SQLite consumers that do not need Sequelize models can use Zacatl's
  `node:sqlite` adapter on Node 26 without installing external `sqlite3` or
  Sequelize.

### Direct UUID Stays Modern

Zacatl's direct UUID dependency remains modern (`uuid@^14`). The audit chain
addressed by this migration is the optional Sequelize subtree, not Zacatl's own
UUID utility.

### Overrides Are Not The Fix

Do not downgrade Zacatl to satisfy npm's suggested fix, and do not rely on a
nested `uuid` override as the primary remediation. The durable fix is to avoid
installing Sequelize in services that do not use SQL persistence.

### DatabaseConfig Uses Named Connections

0.0.57 also standardizes database configuration on `connection.url` with an
optional `connection.name`:

```typescript
// Before 0.0.57
const database = {
  vendor: DatabaseVendor.SQLITE,
  connectionString: 'data/app.db',
};

// 0.0.57+
const database = {
  vendor: DatabaseVendor.SQLITE,
  connection: {
    url: 'data/app.db',
    name: 'default',
  },
};
```

`connection.name` is the stable database token name used by repositories and DI
registration when a service has more than one database of the same vendor. Omit
it for a single database per vendor; Zacatl falls back to the vendor default.

## Upgrade Steps

### 1. Upgrade Zacatl

```bash
npm install @sentzunhat/zacatl@^0.0.57
```

0.0.57 is published; do not point consumers at a future version that has not
been released yet.

### 2. Remove Stale Overrides

If your app tried to fix the advisory with a UUID override, remove it unless you
have a separate application-specific reason to keep it.

```json
{
  "overrides": {
    "uuid": "^11.1.1"
  }
}
```

That override can affect direct dependencies, but it does not reliably remove
Sequelize's nested dependency in the way this migration needs.

### 3. Install Database Peers By Usage

No SQL persistence:

```bash
npm install @sentzunhat/zacatl@^0.0.57
```

MongoDB / Mongoose:

```bash
npm install @sentzunhat/zacatl@^0.0.57 mongoose mongodb
```

PostgreSQL via Sequelize:

```bash
npm install @sentzunhat/zacatl@^0.0.57 sequelize pg
```

SQLite via Sequelize:

```bash
npm install @sentzunhat/zacatl@^0.0.57 sequelize sqlite3
```

Built-in `node:sqlite` adapter:

```bash
npm install @sentzunhat/zacatl@^0.0.57
```

The built-in `node:sqlite` path uses Node 26's runtime module and does not need
the external `sqlite3` package.

For a compact session/archive/training-selection style pattern, see
[`examples/node-sqlite-store`](../../examples/node-sqlite-store/). It shows
explicit migrations, typed row mapping, prepared-statement helpers, display
queries, and cascade delete behavior without Sequelize.

If your app currently uses Sequelize only for a local SQLite file, see
[Migrating Small SQLite Apps From Sequelize To node:sqlite](sequelize-sqlite-to-nodesqlite.md)
for before/after code, dependency-removal guidance, audit expectations, and a
consumer-app checklist.

### 4. Verify The Installed Tree

For a non-SQL service, this should not show Sequelize and should audit cleanly:

```bash
npm ls @sentzunhat/zacatl sequelize uuid
npm audit --omit=dev
```

Expected non-SQL result:

- `@sentzunhat/zacatl@0.0.57` is present.
- No `sequelize` package is installed unless another dependency or the app
  itself installs it.
- Zacatl's direct `uuid` dependency is modern.
- `npm audit --omit=dev` does not report the old
  `@sentzunhat/zacatl -> sequelize -> uuid@8.3.2` chain.

Example non-SQL shape:

```text
your-service
└── @sentzunhat/zacatl@0.0.57
```

For a SQL service, `sequelize` is expected because the app intentionally opts
into the SQL adapter:

```bash
npm ls @sentzunhat/zacatl sequelize uuid
npm audit --omit=dev
```

Example SQL shape:

```text
your-service
├── @sentzunhat/zacatl@0.0.57
└── sequelize@...
    └── uuid@...
```

If npm reports an advisory through `sequelize -> uuid`, treat it as the SQL
consumer's audit surface. The available choices are to keep and document the SQL
adapter risk, move simple local SQLite storage to `node:sqlite`, or upgrade
Sequelize when upstream removes the affected transitive dependency.

Depending on npm's peer-dependency explanation, the audit chain may still list
`@sentzunhat/zacatl` because Zacatl declares Sequelize as an optional peer. In a
SQL fixture where the application explicitly installed `sequelize`, that does
not mean Zacatl core pulled Sequelize into non-SQL consumers.

### 5. Rebuild And Smoke Docker Images

For distroless production consumers, re-run the app's normal release gate after
the dependency update:

```bash
npm test
npm run build
npm audit --omit=dev
docker build .
```

Then smoke the running container with the endpoints that prove the app's real
behavior. For the on-demand-infra runtime, that means at least:

```text
/
/api/ollama/governance/status
```

## Zacatl Example Compose Notes

The Zacatl examples use shared Compose defaults under
`examples/compose/databases/`:

- MongoDB defaults: `examples/compose/databases/mongodb.yml`
- PostgreSQL defaults: `examples/compose/databases/postgres.yml`
- SQLite runtime defaults: `examples/compose/databases/sqlite.yml`

MongoDB and PostgreSQL sidecars default to `mongo:latest` and
`postgres:latest` for local examples. Pin `ZACATL_EXAMPLE_MONGO_IMAGE` or
`ZACATL_EXAMPLE_POSTGRES_IMAGE` before deployment if your environment requires a
specific image digest.

SQLite examples keep the host data bind mount in each example compose file so
the distroless non-root runtime can write the database file safely.

## Checklist

- [ ] App runs Node `>=26.0.0`.
- [ ] App uses `@sentzunhat/zacatl@^0.0.57`.
- [ ] Non-SQL apps do not install `sequelize`.
- [ ] SQL apps explicitly install `sequelize` plus their dialect driver.
- [ ] Stale UUID override removed unless the app has its own reason.
- [ ] `npm audit --omit=dev` passes for non-SQL apps.
- [ ] App tests/build pass.
- [ ] Docker image builds and smokes cleanly.
